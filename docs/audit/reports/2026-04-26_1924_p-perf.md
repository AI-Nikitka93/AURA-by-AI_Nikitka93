## ⚠️ RELIABILITY RISKS
1. `PRF-003` `MEDIUM` `open` — у проекта нет минимального observability-слоя: нет traces, metrics, request id / correlation id и явного server timing. При деградации worker API придётся опираться почти только на локальное воспроизведение. Доказательство: config/code review по `Desaine/worker/index.js`, `Desaine/src/context/ExperienceContext.jsx`, поиск по проекту не нашёл OTel/Sentry/request-id интеграции вне временных `.wrangler` артефактов.
2. `PRF-004` `MEDIUM` `open` — локальный flush очереди заявок работает последовательно и без backoff/jitter. При восстановлении relay очередь либо уходит по одной записи, либо молча остаётся висеть до следующего refresh, что увеличивает риск retry-stall и трудного recovery. Доказательство: `Desaine/src/context/ExperienceContext.jsx:289-323`.

## Perf Summary
- **Perf status:** локальный контур быстрый и стабильно переживает короткий burst на worker API, но публичная главная пока не держит tail latency по `LCP`, а observability остаётся слабым местом.
- **Главная измеренная проблема:** `Public home LCP p95 = 3360ms` при цели `<= 2500ms`.
- **Главный reliability-риск:** отсутствие telemetry + отсутствие backoff/jitter в flush очереди.
- **Главный следующий шаг:** сократить ранний critical path публичной главной, затем добавить хотя бы request id + базовые server/client метрики и повторить тот же perf-harness.

## 🛠️ TOP FIXES
1. Уменьшить ранний critical path публичной главной: self-host или preload для шрифтов, приоритизация только LCP-hero asset, отложенная загрузка вторичных изображений, перенос capability `fetch` за критический рендер.
2. Добавить минимальный observability слой: request id в worker-ответы, structured logs, browser timing export, базовые traces/metrics.
3. Для очереди waitlist добавить ограниченный retry с backoff + jitter и отдельный сигнал о stuck queue.
