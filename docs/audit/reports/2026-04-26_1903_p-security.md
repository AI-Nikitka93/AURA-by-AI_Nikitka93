# SECURITY / PRIVACY / ABUSE AUDIT: AURA by AI_Nikitka93

**Дата:** 2026-04-26 19:03 (+03:00)
**Режим:** STANDARD SECURITY
**Security Score:** 57/100

## Attack Surface
- Публичный web frontend на GitHub Pages: `https://ai-nikitka93.github.io/AURA-by-AI_Nikitka93/`
- Публичный Cloudflare Worker API: `/api/waitlist`, `/api/aura-signal`
- Исходящие интеграции worker: `Resend`, внешний `WAITLIST_WEBHOOK_URL`, `Workers AI`
- Локальное хранение PII и consent в `localStorage`
- GitHub Actions deploy pipeline для Pages
- AI prompt input через публичный JSON API
- Upload/admin/auth surface не обнаружены

## Verified Findings

- Риск: публичный `Workers AI` endpoint фактически не аутентифицирован; защита держится только на `Origin`, который можно просто не отправлять.
- Поверхность: `/api/aura-signal`
- Метод проверки: code review + live runtime probe
- Доказательство:
  - [index.js](/M:/Projects/sites/AURA%20by%20AI_Nikitka93/Desaine/worker/index.js:54) разрешает запросы без `Origin` (`return true`)
  - [index.js](/M:/Projects/sites/AURA%20by%20AI_Nikitka93/Desaine/worker/index.js:368) применяет эту проверку только к `POST /api/aura-signal`
  - live probe на дату аудита: `POST https://aura-portfolio-worker.aiartbora.workers.dev/api/aura-signal` без `Origin` -> `200` и реальная генерация
  - контрольный probe с `Origin: https://evil.example` -> `403`
- Severity: HIGH
- Impact: любой небраузерный клиент может сжигать AI budget и использовать worker как публичный LLM proxy
- Exploitability: HIGH
- Affected surface: public worker AI
- Minimum fix: требовать реальную аутентификацию или антибот-контроль (`Turnstile`/signed token/server secret), отклонять запросы без `Origin`, добавить rate limit и quota
- Статус: CONFIRMED

- Риск: в prompt для модели попадают непроверенные свободные поля клиента, что превращает endpoint в удобный канал prompt abuse и обхода intended flow
- Поверхность: `/api/aura-signal`
- Метод проверки: code review
- Доказательство:
  - [index.js](/M:/Projects/sites/AURA%20by%20AI_Nikitka93/Desaine/worker/index.js:109) валидирует только `language`, `ritual`, `intensity`, `wearMoment`, `ecosystem`, `fitPreference`
  - [index.js](/M:/Projects/sites/AURA%20by%20AI_Nikitka93/Desaine/worker/index.js:179) напрямую вставляет `profileLabel`, `wearLabel`, `ecosystemLabel`, `fitLabel`, `fitNote`, `ecosystemNote`, `localSummary`, `localDirection` в prompt
- Severity: HIGH
- Impact: злоумышленник может использовать endpoint не как product helper, а как общий генератор текста с произвольным payload и дополнительным токен-расходом
- Exploitability: HIGH
- Affected surface: public worker AI / prompt input
- Minimum fix: серверно формировать labels и notes из enum-значений, ввести schema validation + длины + allowlist, отбросить произвольные prose-поля из запроса
- Статус: CONFIRMED

- Риск: в текущем lockfile есть известные уязвимости в toolchain
- Поверхность: dev/build/CI supply chain
- Метод проверки: `npm audit --json`
- Доказательство:
  - `vite@8.0.1` — GHSA-4w7w-66w2-5vf9, GHSA-v2wj-q39q-566r, GHSA-p9ff-h696-f583
  - `postcss@8.5.8` — GHSA-qx2v-qp2m-jg93
  - `picomatch@2.3.1` / `4.0.3` — GHSA-3v7f-55p6-f55p, GHSA-c2c7-rcm5-vvqj
  - CI реально устанавливает эти зависимости в [deploy-github-pages.yml](/M:/Projects/sites/AURA%20by%20AI_Nikitka93/.github/workflows/deploy-github-pages.yml:1)
- Severity: MEDIUM
- Impact: основной риск приходится на dev/preview/CI контур, а не на статический production frontend, но это всё равно подтверждённая supply-chain задолженность
- Exploitability: MEDIUM
- Affected surface: local dev server, preview, CI build agents
- Minimum fix: обновить `vite`, `postcss`, `picomatch` до fixed versions, затем перепройти `npm run build`, `npm run test:visual`, `npm audit`
- Статус: CONFIRMED

  - оценка: HIGH / SUSPECTED

## 🔐 AUTH & ACCESS
- Классического auth/RBAC/admin surface в продукте не найдено; это публичный landing + worker API.
- Поэтому главный access boundary здесь не session management, а защита публичных worker endpoints.
- Для `/api/aura-signal` текущий boundary несостоятелен: `Origin`-check не является аутентификацией.
- Для `/api/waitlist` текущего server-side access control вообще нет; пока это смягчено тем, что public runtime в `device queue`.

## ⚠️ ABUSE CASES
1. `AI cost drain`: любой серверный/bot-клиент без `Origin` может вызывать публичный `Workers AI`.
2. `Prompt abuse`: произвольные prose-поля позволяют использовать endpoint как semi-generic text generator.
3. `Future waitlist spam`: при включении relay публичный `/api/waitlist` будет спам-поверхностью без CAPTCHA/rate limit/auth.
4. `Local PII exposure`: на общем устройстве очередь заявок и email остаются в `localStorage`, пока пользователь вручную не очистит их.

## Security Summary
- **Security status:** есть один реальный HIGH-риск на публичном AI endpoint и один подтверждённый MEDIUM supply-chain долг; до внешнего промо worker API лучше не считать защищённым.
- **Главный подтверждённый риск:** публичный AI endpoint допускает unauthenticated использование при отсутствии `Origin`.
- **Главный plausible risk:** waitlist route станет спам-релеем, если его включить без дополнительных abuse-controls.
- **Главный следующий шаг:** сначала закрыть AI abuse boundary и dependency advisories, потом решать, нужен ли вообще публичный relay для waitlist без server-side anti-abuse слоя.

## 🛠️ MINIMUM FIXES BEFORE RELEASE
1. Закрыть `/api/aura-signal` реальной защитой: signed token / Turnstile / rate limit / quota, а не только `Origin`.
2. Убрать из AI payload свободные текстовые поля клиента или жёстко валидировать и ограничивать их длину; labels формировать на сервере.
3. Обновить `vite`, `postcss`, `picomatch` до фиксированных версий и добавить dependency monitoring.
4. Если планируется включать live waitlist relay, заранее добавить anti-spam контроль: rate limiting, bot challenge, dedupe, возможно HMAC/shared secret для trusted caller path.
